1. PROJECT OVERVIEW
================================================================================

Nexus Machinery Solutions is an industrial machinery repair company based in
Rajkot, Gujarat. They service CNC, VMC, HMC, VTL, and Double Column machines.
The owner wants to transform the business from a traditional phone-call-and-
paper-based operation into a fully automated, paperless digital ecosystem.

The vision encompasses:
  - A customer-facing SERVICE WEBSITE with online booking
  - A customer-facing SELLING WEBSITE for new and used machines
  - An ADMIN PANEL for the owner/manager to control everything
  - A WORKSHOP MANAGEMENT SYSTEM (digital ledger) for the shop floor
  - An EMPLOYEE MOBILE APP for field technicians
  - An AI VOICE AGENT that answers phone calls in Gujarati/Hindi/English

The goal is: COMPLETELY PAPERLESS. Every transaction, every service ticket,
every machine entry, every employee job — all tracked digitally in one
central database.

4. SYSTEM 1: SERVICE WEBSITE (Upgrade of Existing Website)
================================================================================

This is the EXISTING static website that needs to be upgraded. The current
design and look-and-feel will be preserved (red and black theme, Montserrat
font, card-based services layout).

WHAT CHANGES:
  - The WhatsApp wizard will be REPLACED or SUPPLEMENTED with a proper
    booking system
  - A new "Book a Service" section/page will be added

BOOKING SYSTEM — HOW IT WORKS:

  Step 1: Customer visits the website and goes to the services section
  Step 2: Customer selects the type of service they need
          (CNC, VMC, HMC, VTL, Double Column, Component, etc.)
  Step 3: Customer fills a booking form:
          - Name
          - Phone number
          - Location/Address (where the machine is)
          - Machine type
          - Issue description (what's wrong, what service needed)
          - Preferred date/time (from available slots)
  Step 4: System checks available slots from the database
          - Slots are based on employee availability
          - If employee is already assigned somewhere, that slot is blocked
  Step 5: Customer selects an available slot
  Step 6: Booking is confirmed and saved to the database as an "Open Contract"
  Step 7: Customer gets confirmation (on-screen, maybe WhatsApp/SMS)

WHAT THIS CREATES IN THE DATABASE:
  - A new Service Ticket / Contract with status "OPEN"
  - Customer details (name, phone, location)
  - Machine type and issue description
  - Scheduled date and time
  - Assigned slot

ADDITIONAL FEATURES TO CONSIDER:
  - Booking status tracking (customer can check if technician is on the way)
  - Automated reminders (48 hours before service via WhatsApp/SMS)
  - Emergency booking option (priority flag)

7. SYSTEM 4: WORKSHOP MANAGEMENT SYSTEM (Digital Ledger)
================================================================================

This is the PAPERLESS LEDGER system. Currently, the workshop uses physical
registers (ledgers) to track everything. The owner wants to go completely
digital.

THE PROBLEM IT SOLVES:
  The shop receives machines from other companies for repair. Right now, all
  of this is tracked on paper — when the machine came in, what's wrong with
  it, who fixed it, what parts were used, when it was returned. This needs
  to be digitized.

WHO USES IT:
  - The Manager at the shop
  - Could be a web application (desktop browser at the shop) OR a dedicated
    Windows application — TO BE DECIDED

HOW IT WORKS:

  1. MACHINE ARRIVAL (Incoming Entry):
     - A machine arrives at the shop for repair
     - Manager opens the system and creates a new entry:
       * Customer name and company
       * Customer contact number
       * Machine type and model
       * Serial number (if applicable)
       * Issue description (what the customer says is wrong)
       * Date of arrival
       * Photos of the machine (optional)
     - This creates a new "Workshop Ticket" in the database

  2. DURING REPAIR:
     - The assigned employee works on the machine
     - They (or the manager) log what was done:
       * Changed oil
       * Changed oil filter
       * Replaced bearings
       * Did alignment
       * Replaced specific component (e.g., spindle, belt)
       * Any other service performed
     - Each action is recorded with:
       * What was done
       * Parts used (deducted from inventory)
       * Time spent
       * Employee who did it

  3. COMPLETION:
     - When repair is complete, the ticket is marked as "Completed"
     - A summary of all work done is generated
     - This can be used for billing

  4. MACHINE PICKUP:
     - Customer is notified that their machine is ready
     - When they pick it up, the entry is marked as "Delivered"
     - Final billing details are recorded

  ESSENTIALLY: This is the digital equivalent of the physical ledger they
  currently maintain. Every entry, every repair, every part used — all
  recorded digitally and searchable.

8. SYSTEM 5: EMPLOYEE MOBILE APPLICATION
================================================================================

This is a mobile app for the field technicians (employees). When a customer
books a service through the website (or when the AI books it), the employees
see it on their phone and can accept and complete the job.

PLATFORM: Android (primary), possibly iOS later
TECHNOLOGY: To be decided — likely React Native (Expo) for cross-platform

WHO USES IT:
  - All field technicians/employees of Nexus Machinery Solutions

FEATURES:

  1. OPEN CONTRACTS / TICKET DASHBOARD:
     - When the employee opens the app, they see a list of all "Open Contracts"
     - These are service tickets that have been created (by website booking,
       by AI phone booking, or by manager manually)
     - Each ticket shows:
       * Customer name
       * Location/address
       * Machine type
       * Issue description
       * Scheduled date and time
       * Priority level

  2. ACCEPTING A JOB:
     - Any available employee can "Accept" an open contract
     - Once accepted, the ticket is assigned to that employee
     - Other employees can no longer accept it
     - The customer gets a notification: "Your technician is on the way"

  3. NAVIGATION:
     - When the employee accepts, they can see the customer's location
     - Ideally, a map or link to Google Maps for directions

  4. SERVICE LOGGING (On-Site):
     - When the employee arrives and performs the repair, they log everything:
       * Select services performed from a checklist:
         □ Changed oil
         □ Changed oil filter
         □ Replaced bearings
         □ Did alignment
         □ Replaced [component name]
         □ Diagnostics
         □ Calibration
         □ Custom entry (free text)
       * Parts used (select from inventory list — auto-deducts from stock)
       * Photos of the work done (optional)
       * Notes/comments
     - All of this is pushed to the central database

  5. JOB COMPLETION:
     - Employee marks the job as "Completed"
     - The contract/ticket is now closed in the database
     - The database records:
       * Which employee did the job
       * What services were performed
       * What parts were used
       * Time of completion
       * Any notes
     - Customer gets notification that the service is complete

  6. EMPLOYEE PROFILE:
     - Employee's own job history
     - Jobs completed today/this week/this month

10. THE CENTRAL DATABASE — "ONE DATABASE" RULE
================================================================================

ALL systems connect to ONE single central database. This is the backbone
of the entire ecosystem.

WHAT THE DATABASE STORES:

  TABLE/COLLECTION: Customers
  - Customer ID
  - Name
  - Phone number
  - Email (optional)
  - Address/Location
  - Company name (if applicable)
  - Service history (linked tickets)

  TABLE/COLLECTION: Employees
  - Employee ID
  - Name
  - Phone number
  - Skills/specializations
  - Current status (Available / On Job / Off Duty)
  - Job history

  TABLE/COLLECTION: Service Tickets (Contracts)
  - Ticket ID
  - Customer (linked)
  - Machine type
  - Issue description
  - Service type (Repair / Maintenance / Component Change / Breakdown)
  - Status (Open / Accepted / In Progress / Completed / Cancelled)
  - Priority level
  - Scheduled date and time
  - Assigned employee (linked)
  - Location
  - Source (Website Booking / AI Phone / Manual / Walk-in)
  - Services performed (list)
  - Parts used (linked to inventory)
  - Notes
  - Created at
  - Completed at

  TABLE/COLLECTION: Products (Selling)
  - Product ID
  - Name and model
  - Machine type (CNC, VMC, HMC, VTL, Double Column, etc.)
  - Category (New / Used)
  - Price
  - Stock count
  - Images (multiple)
  - Specifications
  - Condition (for used machines)
  - Status (Available / Sold / Reserved)

  TABLE/COLLECTION: Inventory (Spare Parts)
  - Part ID
  - Part name
  - Part number
  - Quantity in stock
  - Supplier details
  - Price
  - Linked to which machine types

  TABLE/COLLECTION: Time Slots
  - Date
  - Time slots with availability status
  - Linked to employee schedules
  - Blocked slots

  TABLE/COLLECTION: Workshop Entries (Digital Ledger)
  - Entry ID
  - Customer details
  - Machine details
  - Date of arrival
  - Assigned employee
  - Repairs performed (list)
  - Parts used (linked)
  - Status (Received / In Repair / Completed / Delivered)
  - Billing details
  - Date of delivery

DATABASE TECHNOLOGY OPTIONS (To Be Decided):
  Option A: PostgreSQL (hosted on Railway or Render)
            + Node.js/TypeScript backend
            - Best for: Complex queries, reporting, relational data
            - Pros: Robust, industry standard, great for structured data
            - Cons: Requires backend setup, more initial work

  Option B: Firebase (Firestore + Authentication + Cloud Functions)
            - Best for: Fast setup, real-time syncing, mobile app integration
            - Pros: No server management, real-time updates, built-in auth
            - Cons: Can get expensive at scale, vendor lock-in, NoSQL

  Option C: Supabase (PostgreSQL + Auth + Storage + Real-time)
            - Best for: Best of both — PostgreSQL with Firebase-like features
            - Pros: Open source, PostgreSQL power, real-time, built-in auth
            - Cons: Newer platform

  NOTE: The owner has mentioned avoiding Supabase. Final decision pending.

11. HOW ALL SYSTEMS CONNECT (Data Flow)
================================================================================

FLOW 1: Customer Books Through Website
  Website → Creates Service Ticket in DB → Visible on Employee App
  → Employee Accepts → Customer gets notification → Employee goes to location
  → Employee logs repair → Ticket closed in DB → Visible in Admin Panel

FLOW 2: AI Books Through Phone Call
  Phone Call → AI understands issue → AI checks slots in DB → AI creates
  Service Ticket in DB → Same flow as Flow 1 after this

FLOW 3: Walk-in Machine at Workshop
  Customer brings machine → Manager creates Workshop Entry in DB
  → Employee assigned → Employee repairs and logs work in DB
  → Machine ready → Customer notified → Machine delivered → Entry closed

FLOW 4: Customer Browses Selling Website
  Customer visits Selling Website → Browses products (data from DB)
  → Sends inquiry → Inquiry stored in DB → Owner sees in Admin Panel
  → Owner contacts customer → Sale happens → Stock updated in DB

FLOW 5: Owner Updates Products
  Owner opens Admin Panel → Adds/edits product listing
  → Uploads images, sets price, stock → Saved to DB
  → Selling Website automatically shows updated data

FLOW 6: Inventory Deduction
  Employee uses a part during repair → Logs part in App
  → App tells DB to reduce inventory count → If stock low, alert in
  Admin Panel → Owner orders more parts
